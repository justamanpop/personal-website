Projects I'm proud of professionally and personally

Professional

  • Framework to run multi-day E2E tests:

    Built from scratch using AWS step functions to orchestrate and manage E2E tests that took more than one day. It included features like polling and waiting for a task to complete, retries, ability to easily add and remove test cases, consolidated reports showing all the steps, CICD, and so on. It taught me the value of constantly building a minimal, working, partial solution first and then refining on that. If I had tried to implement all the aspects the moment I started, I would definitely have failed

  • Optimizing the API CICD pipeline:

    The pipeline ran every time a PR was raised and had to pass before merging to trunk was allowed Run time of the entire pipeline was starting to exceed one hour, and this significantly slowed down development People had lot of idle time waiting for it to finish to merge, and if multiple people wanted to merge, it could take all day just to merge and deploy your changes.

    I introduced parallelism to API test suite runs to reduce the time by 1/6th. Cut down run time from 50ish minutes to around 22 minutes. This really sped up development. Shorter feedback loops, code could be written, tested integrated/deployed that much faster. I genuinely believe it had a good impact on how much we could deliver.

    The second change was to run only a subset of stages in the pipeline based on what files were changed in the PR. For example, a stage related to verifying DB migrations now only ran if migration related files were changed. This cut around 10 minutes from each run, as unnecessary validation was now skipped.

    Technically, it was a good challenge. It involved a bunch of bash scripting to run things in parallel, calculate and parse git diffs, and so on, while alos profiling and grouping tests into approximately even chunks But I'm more proud of identifying a problem we had as a team, and proactively working on it outside my regular assigned work, and showing up with a working solution to ask for permission to implement.

  • Driving refactoring of our UI codebase:

    The UI codebase had some pretty huge, big ball of mud style components that were not testable, frequently caused defects, and were hard to extend or modify without something breaking. Most effort in a sprint was also put into adding features or fixing bugs, refactoring and maintenance were not priorities.

    Starting on the path to fix it was 100% a team effort, and there were people on the team who contributed much more than I did. But I made that initial pitch asking for refactoring time and discussed and helped form a plan of what would be refactored when we did get the bandwidth to do it.

    Since unit tests were mostly not available, helped set up a UI test automation repo and pipeline that tested at the E2E level and got it into reasonable shape to be able to run every day without false failures.This way, there would at least be some cover telling us when we broke something when refactoring

    We did eventually get bandwidth to refactor, and made a lot of progress. There's a long way to go, but the codebase is definitely healthier than it was, and a system is in place to enable further refactoring to happen

I made a simple mobile app that at a configurable interval and duration, flashed with a different color, and based on the color, the player is supposed to do something different with the incoming ball.
